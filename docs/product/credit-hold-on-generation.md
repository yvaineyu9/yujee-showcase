# 生成相册的积分锁定 · 结算需求

> 产品经理产出。**这是独立于「UI 还原 gap」的另一块需求**——那块是修页面(前端视觉),这块是积分账务的事务逻辑(后端计费正确性)。两者刻意分开,互不混。
> 本文只定**产品规则**(什么时候锁、什么时候扣、什么时候退),不定技术实现(怎么接、放哪个 service)——那是架构的事。
> 本文基于实读 `apps/web/src/services/credit/` + `app/api/v1/jobs/` 现状。

---

## 一句话问题

用户生成相册时,**没有积分校验、没有锁定、没有扣减**——余额为 0 也能发起生成,生成失败了账务也不对。需要把"先锁定、成功才扣、失败退回"的逻辑接上。

## 给谁用

所有发起生成的注册用户(消耗积分的核心动作)。

## 现状(实测,关键)

**账务零件已经全部造好,只是没装到生成流程上:**

| 已存在 ✅ | 位置 |
|---|---|
| `precheck(userId, amount)` 余额校验,不足抛 `CREDIT_INSUFFICIENT` | `services/credit/credit-service.ts` |
| `hold(...)` 锁定:建 `credit_holds` 行 + 挂到 `albums.credit_hold_id`,余额不变,落 0-delta 账本痕迹 | 同上 |
| `settle(tx, userId, holdId)` 结算:实扣余额、关闭 hold | 同上 |
| `release(holdId)` / `releaseByAlbum(albumId)` 释放锁定 | 同上 |
| `CREDIT_COST_PER_ALBUM` 单本相册成本常量(**金额已定义,本需求不改它**) | credit-service 引用 |
| `credit_holds` 表 + ledger 的 `GENERATION_HOLD/SETTLE/RELEASE` 理由 | `db/schema.ts`、migrations |
| `zombie-sweeper` 卡死任务清扫器(可用于释放悬挂锁定) | `services/job/zombie-sweeper.ts` |

**缺口(就是本需求要补的):**
- ❌ 生成入口 `POST /api/v1/jobs`(`app/api/v1/jobs/route.ts`)**完全没调积分**——没 precheck、没 hold。
- ❌ 生成完成/失败的状态流转(`generation-service` 等)**没在结束时 settle / release**。
- 即:**precheck/hold/settle/release 这套原语都现成,但没被生成生命周期调用。**

## why now

订单积分扣减这块"还有很大问题"(用户原话)。已上线、用户在真实消耗积分,账务不接 = 漏钱 + 失败误扣,属于上线后必须尽快补的计费正确性问题。

## 目标逻辑(最小可行)

**核心流程规则(用户确认)**:**任务在用户发起时就创建,但 agent 执行被"积分校验通过"这道闸挡住。** 即——发起 ≠ 立即跑 agent;先建任务(带着 prompt + 照片),等积分这关过了(余额够 或 付完款)才真正调 agent 执行。这样付款回来后天然能接着原任务跑,无需让用户重填。

按"建任务 → 校验闸 → 锁定 → 按结果结算"展开:

1. **用户发起生成(提交 prompt + 照片)**
   - **立刻创建任务/相册记录**(占位,带原始 prompt + 照片),**但先不调 agent**。
2. **积分校验闸(调 agent 之前)**
   - `precheck`:余额 ≥ `CREDIT_COST_PER_ALBUM`?
     - **足够** → `hold` 锁定成本(余额此刻**不变**,只冻结)→ 放行,**触发 agent 执行该任务**。
     - **不足** → 拦住,不跑 agent,**跳转现有 `/pricing` 页**(那页本就同时有"开会员/订阅"和"买积分/充值包",不另造)。用户付完款、余额够了 → `hold` → **回到刚才那个已存在的任务继续执行**。
3. **生成结果回来时**
   - **成功(COMPLETED)** → `settle`:把锁定额度**实扣**,余额减少,hold 关闭。**扣费只发生在成功这一刻。**
   - **失败** → **不立即扣、也不立即退**,**保持锁定**,让用户可**免费重试同一任务**(复用同一 hold,不重新计费),**直到成功才 settle**。
   - 用户**放弃/任务卡死**导致的悬挂锁定 → 由"释放策略"兜底回收(见待决策 3)。

> 核心承诺:**任务先建、扣费最后;只在生成成功时扣一次;失败可免费重试,成功前一分不扣。**

## 成功标准(可度量)

- 余额 < 单本成本 → **不触发 agent**(任务可建,但被校验闸挡住,引导付款)。
- 生成失败 → 用户重试**不重复计费**;失败到放弃为止**不扣分**。
- 生成成功 → 恰好扣一次 `CREDIT_COST_PER_ALBUM`,不多不少。
- 付款成功后 → 自动接回原任务(原 prompt + 照片),用户不用重来。
- **不存在永久悬挂的 hold**(放弃/卡死的锁定最终被释放)。

## 不做(scope 边界,显式)

- 不改定价、不改 `CREDIT_COST_PER_ALBUM` 数值。
- 不做编辑器里二次修改 / 重新排版的计费(本需求只管**首次生成**这一次锁定+扣减)。
- 不做退款 / 发票逻辑(那是 billing 域,另说)。
- 不碰 UI 视觉还原(那是另一份 `ui-restore-gap.md`)。

## 决策状态

### 已定 ✅(用户确认)
1. **余额不足拦截**:校验在**调 agent 之前**;不足 → **跳转现有 `/pricing` 页**(不另造,那页已含开会员 + 买积分),付完接回原任务。
2. **失败重试不重新计费**:复用同一 hold,失败可免费重试,**直到成功才扣费**。
3. **任务创建与 agent 执行解耦**:发起即建任务(占位),agent 执行等积分校验过了才触发——这也使"付款后接回原 prompt/照片"天然成立。

### 仍需你 + 架构定 🟡
1. **悬挂锁定的释放策略**:用户放弃/任务卡死时,锁定的积分**多久、由谁释放**?(你说要和架构讨论。)产品上只硬性要求"最终一定释放,不能永久占用";是否复用现成的 `zombie-sweeper` → `releaseByAlbum` 交给架构。
